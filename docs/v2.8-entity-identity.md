# V2.8 · Entity Identity Law（实体身份法则）

> **状态**：✅ 已落地（Phase C）；Phase D **D3 Entity Resolver** 为全系统消费者入口。  
> **重要性**：与 [v2.8-measurement-layer.md](./v2.8-measurement-layer.md) 同级；Knowledge 为唯一事实源时，**身份**比字段更重要。  
> **实现**：`story-schemas/entity-id.js` · `knowledge/entities/*` · `migration/id-mapping.json`

---

## 一、要解决什么问题

同一角色可能同时来自：

```text
正文扫描 · 设定集 · 用户手动 · Understanding 推断 · 旧 studio.assets
```

若 `id = name`（**当前实现**），用户将「欧阳逸清」改名为「顾长歌」时：

```text
所有关系、伏笔引用、Actions、Plans、Versions diff 全部断裂
```

Entity Identity Law 定义：**实体如何在全系统内保持同一身份**。

---

## 二、架构法则 6（写入 architecture.md）

### 实体 ID 与显示名分离

| 字段 | 可变性 | 用途 |
|------|--------|------|
| **`id`** | **不可变**（创建后永不改） | 全系统唯一引用键 |
| **`name`** | 可变 | 当前显示名 |
| **`aliases`** | 可变 | 曾用名、扫描命中、笔名 |
| **`slug`** | 可选、可变 | 导出/URL 友好名，**非**引用键 |

```text
引用规则：代码、JSON、Plan、Action、Relationship 一律使用 id，禁止用 name 作外键。
```

### 跨层引用

| 消费者 | 引用方式 |
|--------|----------|
| Knowledge relationships | `from_id` / `to_id` |
| Understanding arcs | `character_id`（非姓名） |
| Actions / Plans | `entity_refs: [{ type, id }]` |
| Measurement / Verify | 不涉及实体合并（只读 stats） |
| Memory / RAG | 元数据带 `entity_id`，重建时可丢 |
| 全系统解析 | `story-kb/entity-resolver.js` · `resolveMany()`（D3.6） |
| Versions diff | 路径级，不绑实体 id |

## 三、ID 生成策略（定稿）

### 3.1 格式（CTO 签字：**base32(8)**，非 hex）

```text
{type_prefix}_{base32_8}

示例：
  char_a3f9k2m1
  loc_b7c2x9p0
  org_c1d4e8f2
  rel_r8h3j5k7   # 关系边可选独立 id
```

字母表：小写 `0-9` + `a-z`（实现见 `story-schemas/entity-id.js`）。**禁止** 8 位纯 hex。

| type_prefix | 实体类型 |
|-------------|----------|
| `char` | 角色 |
| `loc` | 地点 |
| `org` | 组织/势力 |
| `rel` | 关系边（若独立存储） |
| `evt` | 时间线事件（若独立 id） |

**生成时机：** 实体**首次进入 Knowledge 持久化**时分配（用户创建、迁移导入、pipeline 确认入库）。

**禁止：** 用姓名 hash、slug、UUID v4 裸存无类型前缀、纯 hex 后缀（不利于调试与迁移审计）。

### 3.2 与现状（LEGACY）的关系

| 现状 | 问题 | Phase C 迁移 |
|------|------|----------------|
| `id: "欧阳逸清"` | id=name | 分配 `char_*`，`name` 保留，`aliases` 含旧名 |
| 噪声 id「我知」「个巨大的」 | 扫描误识别 | 标记 `status: rejected` 或删除，不分配稳定 id |
| `source: inferred` 且无 user 项 | 易覆盖 | 见 §五 合并规则 |

迁移脚本必须为每个保留实体生成**新** `id`，并写 `legacy_id` 可选字段记录原 name-key。

## 四、生命周期操作

### 4.1 创建（Create）

```text
1. 用户或 pipeline 提交候选（name, source, …）
2. 查 aliases 索引 + 模糊匹配（见 §4.5）
3. 若命中已有实体 → 走合并/更新，不新建 id
4. 否则 generateId(type) → 写入 entities/{type}.json
```

**Pipeline（inferred）创建：** 仅当无 `source=user` 同 name/alias 命中时可新建；默认 `status: provisional`，用户确认后升为 `active`。

### 4.2 改名（Rename）

```text
禁止：修改 id
必须：
  - 更新 name
  - 将旧 name 推入 aliases（若不在 aliases 中）
  - 写 audit：{ at, op: "rename", from, to, by: "user"|"import" }
```

正文与设定集 MD **不必**自动改名；Knowledge 的 `name` 是 Story Assets 显示名，与文稿可暂时不一致（由 Action 提示对齐）。

### 4.3 合并（Merge）

用户确认「欧阳逸清」与「逸清」为同一人：

```json
{
  "id": "char_a3f9k2m1",
  "name": "欧阳逸清",
  "aliases": ["逸清", "欧阳"],
  "merged_ids": ["char_deprecated_x"],
  "status": "active"
}
```

被合并实体：

```json
{
  "id": "char_deprecated_x",
  "status": "merged",
  "merged_into": "char_a3f9k2m1",
  "merged_at": "..."
}
```

**规则：**

- `merged` 实体不可再被引用；读取时 resolve 到 `merged_into`
- relationships 重指向 target_id
- 禁止物理删除有引用的实体（见 §4.4）

### 4.4 删除（Delete）

| 类型 | 策略 |
|------|------|
| 用户删除 | `status: deleted` + `deleted_at`；保留 id 占位 |
| 误识别噪声 | `status: rejected`；可从列表隐藏 |
| 硬删 | **禁止**（破坏引用与版本审计） |

### 4.5 去重与匹配（Match）

匹配优先级（高 → 低）：

```text
1. 精确 id 引用
2. aliases 精确匹配
3. name 精确匹配（仅当无同名冲突）
4. pipeline 模糊匹配 → 产出 merge 建议 Action，不自动合并
```

**禁止：** Understanding 自动 merge 两个 `source=user` 实体。

## 五、写入冲突（与公约 4 衔接）

```text
user PUT（带 id）        >  user PUT（按 name 更新已有 id）
user 创建/更新           >  pipeline provisional
pipeline 新建            >  不得覆盖 user 实体字段
Understanding 分析       >  不得写 entities；只写 understanding/
```

**`source` 字段（保留）：** `user` | `inferred` | `import` | `migrated_from_studio`

```json
  "name": "欧阳逸清",
  "aliases": ["逸清"],
  "slug": null,
  "role": "protagonist",
  "notes": "",
  "source": "user",
  "status": "active",
  "created_at": "2026-06-02T00:00:00.000Z",

  "merged_into": null,
  "legacy_id": null
}
```
