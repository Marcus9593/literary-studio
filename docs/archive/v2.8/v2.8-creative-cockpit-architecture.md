## 为什么需要本文

在「目标产品形态」未定时写数据迁移，会导致：

```text
studio.json assets
  → knowledge/
  → story-kb/
  → Story OS 某层
```

反复重迁。当前最缺的是 **Creative Center 最终形态**，而非迁移代码。

## 四层模型（定稿）

不是「Workspace + Story OS + Creative Center」三块，而是四层，各答一个维度：

| 层 | 英文名 | 动词 | 一句话 |
|----|--------|------|--------|
| **1** | Workspace | 创造 | 写东西 |
| **2** | Story OS | 推进 | 作品往哪走、今天做什么 |
| **3** | Creative Cockpit | 监控 | 全库健康度、进度、产能、路线（跨项目 Dashboard） |
| **4** | Story Assets | 管理 | 角色、地点、组织、设定、时间线、关系（= Knowledge） |

```text
┌─────────────────────────────────────────────────────────┐
│  Workspace          创造内容（正文 / 大纲 / 对话 / 执行）   │
├─────────────────────────────────────────────────────────┤
│  Story OS           推进内容（Goal / Roadmap / Tasks /     │
│                     Plans / Actions / Health）            │
├─────────────────────────────────────────────────────────┤
│  Creative Cockpit   监控内容（跨项目 Dashboard）           │
├─────────────────────────────────────────────────────────┤
│  Story Assets       管理资产（Knowledge，项目内）          │
└─────────────────────────────────────────────────────────┘
```

**全局 `/studio` 不叫「第四套故事功能」**，而是 **创作管理中心**：只放跨项目、非编辑、非单项目推进的两件事。

## 问题 1 · Creative Center 最终保留什么？

### 答案：仅两项（两个全局 Tab）

| 保留 | 系统名 | 用户可见名 | 职责 |
|------|--------|------------|------|
| ✅ | **Creative Cockpit** | 创作驾驶舱 | 跨项目监控：健康度摘要、进度、产能、路线、待办 Task |
| ✅ | **Versions** | 项目版本 | 跨项目版本库：快照、Diff、恢复、历史（按项目筛选） |

```text
Creative Center（全局 /studio）
├── Creative Cockpit    默认首页
└── Versions            版本中心
```

### 明确不保留在 Creative Center

| 不保留 | 原因 |
|--------|------|
| 素材中心 | 属于 Story Assets（Knowledge），项目内管理 |
| 审稿中心 | 属于 Health 能力，非独立中心 |
| 作品体检（独立 Tab） | 详情在项目内 Health；Cockpit 只展示**摘要/跳转** |
| 旧「写作看板」四指标主展示 | 并入 Cockpit 的「产能/活跃」区块，非独立产品 |

### Creative Cockpit 监控维度（对标 Notion / Linear / GitHub Insights）

| 区块 | 内容示例 | 数据源 |
|------|----------|--------|
| 产能 | 近 7 日活跃、总字数（可折叠） | `meta.updated_at`、项目 stats |
| 进度 | 正在推进 / 停更风险 / 已完成 | `planner/`、`tasks/`、启发式 |
| 健康度 | 各项目分数条、最低分预警 | Story OS **Health** 聚合 |
| 路线 | 待完成 Task、下一章 Roadmap 摘要 | `tasks/tasks.json`、`chapter_roadmap.json` |

Cockpit **不**承载：写稿、改 KB、跑审稿、编辑快照内容。

### Versions 职责

| 能力 | 说明 |
|------|------|
| 快照列表 | 按项目筛选，全库时间线可选 |
| Diff / 恢复 | 保留现有 `studio-service` 行为，目标目录 `versions/` |
| 元数据 | `type`（manual / rewrite / auto_write）、`parent`（版本树，UI 可后置） |
| 自动保存 | 写稿完成事件 → 写入 Versions（不变） |

## 问题 2 · 哪些能力迁出？

### 答案

```text
Assets  →  Knowledge（Story Assets，项目内）
Review  →  Health（Story OS 子能力，非独立中心）
```

### Assets → Knowledge

| 现状 | 目标 |
|------|------|
| `data/studio.json` → `assets[projectId][]` | **废弃**为写入源 |
| Studio「素材中心」Tab | **删除** |
| 用户心智 | 「素材 = 项目知识库里的实体」 |

**Story Assets（Knowledge）范围：**

```text
角色 · 地点 · 组织 · 设定 · 时间线 · 关系
```

实现载体：`data/projects/{id}/knowledge/*.json`（及 story-kb 同步），**不再**新增 `studio.assets` 字段。

> **边界说明**：`story-kb` 是 KB 的**构建/索引管道**，不是用户可见的第四层；用户只认 **Knowledge（Story Assets）**。

### Review → Health

现状三处打架：

```text
Studio 审稿中心     → 启发式四维分 + hints
项目 Health 页      → verify + health_snapshot
Actions             → 诊断建议（为什么）
```

**目标链：**

```text
Health（用户入口）
    ↓
Review Engine（启发式 + 未来 LLM，内部模块）
    ↓
Actions（诊断输出，source: review | verify | understanding）
```

| 概念 | 归属 |
|------|------|
| 「重新体检」按钮 | 项目 `/health`，触发 Review Engine |
| 节奏 / 人物 / 冲突 / 伏笔 / 风格 | `health/score.json` |
| 趋势 | `health/trends.json` |
| 诊断文案 | `health/diagnostics.json`（合并 hints + actions 摘要） |
| `studio.review_by_project` | 迁移源 only，**非**长期真相源 |

**Cockpit 与 Health 关系：** Cockpit 只读各项目 `health/score` 做排行榜/预警；深度体检进项目 Health，不在 `/studio` 开第三 Tab。

## 问题 3 · 最终导航结构

### 项目内（`/projects/:projectId`）

```text
项目工作台
├── Workspace          /projects/:id          正文 · 大纲 · AI 对话 · 计划执行
├── Story OS           （子导航）
│   ├── 今日创作       /suggestions          Tasks · Scheduler
│   ├── 路线图         /roadmap               Roadmap · Goal（路由待建）
│   ├── 修改计划       /plans                 Plans
│   └── （诊断）       并入 Actions/Health，不单列「诊断中心」
├── Knowledge          /knowledge             Story Assets（含原 studio 素材）
├── Health             /health                体检 · Review Engine · Verify 聚合
└── Plans              与上重复时以「修改计划」为准，Roadmap 独立后调整子 nav
```

**Story 子导航目标态（收敛后）：**

```text
今日创作 · 路线图 · 修改计划 · 知识库 · 健康度
```

（「作品知识」文案统一为 **知识库**；素材不再单独出现在全局 Studio。）

### 全局（`/studio`）

```text
创作管理中心（Creative Center）
├── Creative Cockpit   /studio 或 /studio/cockpit
└── Versions           /studio/versions
```

### 顶层侧栏（App 级）

```text
项目库          /
创作管理中心     /studio     → Cockpit + Versions only
工具中心         /tools
AI 配置          /settings
```

### 与旧 Studio 四 Tab 对照

| 旧 Tab | 归宿 |
|--------|------|
| 写作看板 | → Creative Cockpit（重构指标） |
| 版本与快照 | → Versions |
| 审稿中心 | → 项目 Health（Review Engine） |
| 素材中心 | → 项目 Knowledge |

## 问题 4 · 迁移路线

**原则：** Phase C 之前不写 `migrate*`；Phase B 只做能力归位（路由/API 只读、deprecated 标记）。

```text
Phase A   冻结              P0 ✅  禁止 studio assets/review 新功能
Phase B   能力归位          P2 🚧  目标架构本文 + 更新 roadmap/API 契约
Phase C   数据边界 + 迁移   P3–P4  定 health/*、knowledge 边界后一次性迁移
Phase D   UI 重构           P5     Cockpit + Versions 两 Tab；删 Studio 旧 Tab
```

### Phase A · 冻结（已完成）

- [x] `roadmap-v2.8-creative-center.md` P0
- [x] `studio-service` / `StudioPage` deprecated 注释
- [x] 禁止迁移脚本

### Phase B · 能力归位（当前应做）

| 任务 | 产出 | 不做 |
|------|------|------|
| 采纳本文为目标架构 | 评审锁定四问题答案 | 改 Studio UI |
| 更新 `architecture.md` §十二 | 四层模型 + 仅 Cockpit/Versions | 双写 knowledge |
| API 契约草案 | `GET /studio/cockpit`、`GET /studio/versions` 字段说明 | `migrateStudioAssets*` |
| Health 契约草案 | `health/score|trends|diagnostics` schema | 删旧 review API |
| 标记废弃路由 | `/studio/assets*`、`/studio/review*` → deprecated | 物理删路由（可 Phase D） |

### Phase C · 数据边界 + 迁移（Phase B 锁定后）

1. **定边界文档** ✅ [v2.8-data-boundaries.md](./v2.8-data-boundaries.md) · [v2.8-measurement-layer.md](./v2.8-measurement-layer.md)  
2. **一次性迁移**（单脚本、可回滚）：  
   - `studio.assets` → `knowledge/*`  
   - `studio.review_by_project` → `health/*`  
   - `studio.snapshots` → `projects/{id}/versions/`  
3. 迁移后 `studio.json` 仅保留兼容读或清空并归档。

### Phase D · UI 重构

- `StudioPage` → 两 Tab：Cockpit、Versions  
- 项目 Knowledge 增加/强化实体 Tab（角色/地点/设定…）  
- 项目 Health：「重新体检」、五维分数、趋势  
- Cockpit 卡片链到项目 Health / Tasks
