## 页一 · Entity Migration Policy

### 1.1 迁移分类（只三类）

| 类别 | 写入 `entities/` | 分配 `char_*` | 进入 `reject_report` |
|------|------------------|---------------|----------------------|
| **ACTIVE** | ✅ `status: active` | ✅ | 否 |
| **PROVISIONAL** | ✅ `status: provisional` | ✅ | 否（附 `reason`） |
| **REJECTED** | ❌ | ❌ | ✅ |

```text
宁可 PROVISIONAL，不要误删真角色（小白、阿七、叶秋保留路径）。
禁止 REJECTED 仍分配稳定 id — 否则噪声被 Actions/Plans 永久引用。
```

### 1.2 判定规则（按优先级）

| 规则 | 条件 | 结果 |
|------|------|------|
| **R1** | `source === 'user'`（或迁移后用户确认） | **ACTIVE** |
| **R2** | 被 `relationships` 任一边引用（by id 或 legacy name） | **ACTIVE** |
| **R3** | 正文出现次数 ≥ **3**（全项目 manifest 扫描，按 `name` 匹配） | **ACTIVE** 候选；=1–2 且非 R1/R2 → **PROVISIONAL** |
| **R4** | 命中**纯停用词表**（见下） | **REJECTED** |
| **R5** | 命中**明显截断/扫描碎片**（见下） | **REJECTED** |

**实现顺序**：R1 → R2 → **R4 → R5** → R3 → R6。R4/R5 必须在 R3 之前，否则「我知」等会因子串命中（如「我知道」）被误升为 ACTIVE。
| **R6** | 其余 `source=inferred` | **PROVISIONAL** |

**不用**单独 `length < 3` 作为拒绝条件。

#### 停用词表（R4，可扩展）

```text
我知, 那个人, 巨大的, 个巨大的, 阳逸清没, 他说, 她说, 众人, 有人
```

#### 截断碎片（R5）

满足任一 → **REJECTED**：

- 名称长度 ≤ 2 **且** 非 R1/R2/R3  
- 含明显 OCR/分词错误模式（连续虚词、无中文姓特征的单片段）  
- Inventory 已列样本：`我知`、`阳逸清没`、`个巨大的` → **REJECTED**

#### 示例（583e5628-24b）

| 原名 (legacy id) | 判定 | 新 id |
|------------------|------|-------|
| 欧阳逸清 | ACTIVE（R3 正文高频） | `char_{base32_8}` |
| 我知 | REJECTED（R4/R5） | — |
| 阳逸清没 | REJECTED（R5 截断） | — |
| 个巨大的 | REJECTED（R4/R5） | — |

### 1.3 迁移写入形状

**ACTIVE / PROVISIONAL** 写入 `knowledge/entities/characters.json`：

```json
{
  "id": "char_a3f9k2m1",
  "entity_type": "character",
  "name": "欧阳逸清",
  "aliases": [],
  "legacy_id": "欧阳逸清",
  "source": "inferred",
  "status": "active",
  "migration": { "at": "...", "rule": "R3", "from_file": "characters.json" }
}
```

**REJECTED** 仅写入 `migration_report.json` → `rejected_entities[]`，**不**进入 entities。

### 1.4 其他 knowledge 文件（本阶段）

| 文件 | Phase C 动作 |
|------|----------------|
| `locations`, `relationships`, `timeline` | 平移 → `entities/*`；relationship 边改为 `from_id`/`to_id`（Dry Run 输出映射表） |
| `foreshadows.json` | 迁 `understanding/foreshadows.json`，**不**留 entities |
| `story_summary.json` | logline 等 → `understanding/story_dna` 或 `knowledge/notes/imported_summary.json`（Dry Run 定一条路径） |

## 页二 · Snapshot Migration Policy

### 2.1 原则（CTO 签字）

```text
Version 必须保留内容快照，禁止「仅 path 引用」替代历史副本。
否则 workspace 变更后，旧版本不可恢复，违背 Version 语义。
```

### 2.2 目标布局

```text
data/projects/{project_id}/versions/
├── manifest.json              # 版本列表 + 当前 head
└── {version_id}/              # 如 v_f72c7b2e6e0
    ├── metadata.json          # title, type, parent, created_at, source_snapshot_id
    └── files/                 # 与 workspace 相对路径一致
        ├── 正文/第0003章-….md
        └── 大纲/总纲.md
```

- `metadata.json` **不**内嵌 `files[].content`（避免单文件膨胀）  
- `files/` 下为迁移时从 snapshot **完整拷贝**的 UTF-8 文本（与现 studio 行为等价）

### 2.3 体积决策（Inventory 实测）

| 指标 | 值（2026-06-02） |
|------|------------------|
| `studio.json` 文件 | **0.24 MB** |
| 快照内 `content` 合计 | 0.23 MB |
| 跨快照去重后唯一内容 | 0.08 MB（理论省 **64.7%**） |

**结论（当前规模）：**

```text
采用「每版本完整 files/ 副本」— 实现简单，0.24MB 可接受。
不做跨版本 content 去重（复杂度 > 收益）。
```

**再评估阈值（写入 conventions，非本次实现）：**

| 条件 | 策略 |
|------|------|
| 单项目 `versions/` < 50 MB | 完整副本 |
| 单项目 versions ≥ 50 MB | 评估 block 级去重或压缩（Phase C+1） |

Dry Run 必须输出：`snapshot_bytes_before` / `snapshot_bytes_after` / `dedup_hypothetical_savings`（见 §3）。

### 2.4 从 `studio.snapshots` 迁移步骤

1. 每条 legacy snapshot → 一个 `version_id`（可沿用原 `id` 或 `v_{id}`）  
2. 复制 `files[].content` → `versions/{id}/files/{path}`  
3. `metadata.json` 保留 `title`, `notes`, `created_at`, `parent: null`（ lineage 后续再加）  
4. 迁移完成后 `studio.snapshots[projectId]` **只读归档**，新写入走 `versions/`

## 三 · Dry Run 规范（2C-2，不写盘）

### 3.1 命令（规划）

```bash
node backend-node/scripts/migration-dry-run.mjs --project <id> --out reports/
```

- **禁止** 修改 `data/projects` / `data/studio.json`  
- 输出仅 JSON 报告 + stdout 摘要

### 3.2 输出文件

| 文件 | 用途 |
|------|------|
| `migration_report.json` | 将发生什么（映射、计数、体积） |
| `reject_report.json` | 被拒绝实体及规则命中 |

Schema：[migration-report.schema.json](./schemas/migration-report.schema.json) · [reject-report.schema.json](./schemas/reject-report.schema.json)

### 3.3 `migration_report.json` 最小字段

```json
{
  "schema": "migration_dry_run",
  "project_id": "583e5628-24b",
  "generated_at": "…",
  "entities": {
    "active": 1,
    "provisional": 0,
    "rejected": 3,
    "id_mappings": [{ "legacy_id": "欧阳逸清", "new_id": "char_x", "status": "active", "rule": "R3" }]
  },
  "snapshots": {
    "count": 3,
    "bytes_before": 250432,
    "bytes_after_estimate": 240000,
    "dedup_hypothetical_savings_pct": 64.7,
    "policy": "full_content_per_version"
  },
  "measurement": { "review_migrate": true, "verify_log_migrate": true, "health_snapshot_drop": true },
  "warnings": []
}
```

### 3.4 `reject_report.json` 最小字段

```json
{
  "schema": "migration_reject_report",
  "project_id": "583e5628-24b",
  "rejected_entities": [
    { "legacy_id": "我知", "name": "我知", "type": "character", "rule": "R4", "reason": "stopword_fragment" }
  ]
}
```

### 3.5 Dry Run 通过标准（Phase C 前门）

- [x] `rejected_entities` 无应保留项（我知 / 阳逸清没 / 个巨大的 — REJECTED 可接受）  
- [x] ACTIVE 唯一映射（欧阳逸清 · R3 · mentions=320）  
- [x] snapshot `bytes_after_estimate` 可接受；策略 `full_content_per_version`  
- [x] **CTO 签字**（2026-06-02）：Reject Report · Dry Run · Snapshot · Measurement · Entity Identity  
- [x] [Rollback Plan](./v2.8-phase-c-rollback.md) 已补（Phase C 前置）
