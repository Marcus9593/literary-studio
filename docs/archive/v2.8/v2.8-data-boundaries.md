## 一、为什么要先有本文

Narrative IDE 的致命错误不是 Tab 多一个，而是：

```text
角色资料到底放哪？
```

若未封死边界，会出现 Knowledge、Story KB、Understanding、Memory、Planner Context 五套「好像都能放角色」的系统，且 **Planner 自动写回 KB** 后真相源崩溃。

本文用一张矩阵回答：**谁写、谁读、用户能否编辑、属于哪一层**。

## 二、架构公约 4（写入 architecture.md）

### Knowledge 是唯一用户可编辑的故事事实源（Story Assets）

| 类型 | 用户可编辑？ | 说明 |
|------|-------------|------|
| **Knowledge**（`knowledge/*.json`） | **是** | 角色、地点、关系、时间线、伏笔、摘要等实体 |
| **Workspace**（`workspace/` 文稿） | **是** | 叙事正文、大纲、设定集 Markdown — 创作真相源 |
| **Plans**（`plans/*.json`） | **是** | 通过确认/执行/完成流程编辑 |
| **Planner preferences** | **是** | `planner/preferences.json` |
| Story KB（模块名） | **否** | 不是存储层，见 §三 |
| Story Index | **否** | 内存索引，由 Knowledge 重建 |
| Understanding | **否** | 引擎产出；用户只能 **触发重建**，不能直接改 JSON |
| Goal / Roadmap / Tasks | **受限** | 由 Planner 生成；用户改 **偏好** 或 **触发 regen**，不手改队列文件（V3 可加审计式修正） |
| Actions | **否** | 诊断输出；用户 **采纳** → Plan |
| Health / Verify | **否** | 测量结果；用户 **重新体检** |
| Memory / Vectors | **否** | 会话与检索缓存 |
| Planner Context | **否** | 运行时组装的 prompt 块，不落盘为真相源 |
| `studio.json` assets/review | **否（废弃）** | 仅迁移源 |

**Planner / Claude 禁止（V2.8+）：**

```text
禁止 Planner 自动覆盖 Knowledge 实体字段（name、role、关系等）
禁止 Understanding 在无用户触发时批量覆写已有 Knowledge（source=user）
禁止把 Understanding 当「第二套可编辑设定集」暴露给 UI
```

**允许（须带 `source` 与合并规则）：**

```text
bootstrapFromWorkspace / quick_sync 可追加 source=inferred 的 KB 项
用户 PUT /story/knowledge 始终可覆盖同 id 实体
```

## 三a、Knowledge 拆分：Story Assets ≠ Story Intelligence（炸点 1）

Knowledge 当前混有两类职责，迁移前须在架构上先拆开：

| 类型 | 性质 | 目标路径 | 示例 |
|------|------|----------|------|
| **Story Assets** | 用户可编辑的**事实** | `knowledge/entities/*` | 角色苏晚、地点、组织、关系 |
| **用户备注** | 用户笔记 | `knowledge/notes/*` | user_notes、imported_notes |
| **Story Intelligence** | 引擎**推断** | `understanding/*` | 伏笔、悬念、冲突、弧光、主题 |

**迁出 Knowledge（禁止长期双写）：**

- `foreshadows.json` → `understanding/foreshadows.json`
- `story_summary` 中推断型 logline → Understanding 或 `notes/`，不当实体表

**引擎规则：** Understanding / Review **不得**把推断结论写回 `knowledge/entities`；只可写 `understanding/`。  
`syncForeshadowsToKb` 等现有写回路径在 Phase C 前标记 deprecated，迁移后删除。

---

## 三b、Story KB ≠ Knowledge（封死混淆）

| 名词 | 是什么 | 不是什么 |
|------|--------|----------|
| **Knowledge** | `data/projects/{id}/knowledge/*.json` — **用户可见、可编辑**的 Story Assets | 不是模块名 |
| **story-kb/**（代码包） | **构建与索引管道**：`sync.js`、`scan-sources.js`、`rebuild.js`、`store.js` | **不是**用户可见的第四层 |
| **Story Index** | `story-index/build.js` 内存 Map，Knowledge PUT 后 `rebuildStoryIndex` | 不是持久化真相源 |

```text
Workspace Markdown  ──scan──►  story-kb pipeline  ──write──►  knowledge/*.json
                                      │
                                      └──►  story-index (cache)
                                      └──►  understanding engine (read KB, write understanding/)
```

**对用户只说「知识库」**，不说「Story KB」。

## 四、Measurement Layer 与 Health（UI 名）

> **专篇**：[v2.8-measurement-layer.md](./v2.8-measurement-layer.md) — Review / Verify / Metrics / Trend / Health 五者关系（**当前最高架构风险**）。

```text
measurement/          # 存储层（禁止用 health/ 当存储根）
├── review/           # Review Engine
├── verify/           # 验收
├── metrics/          # 分数与维度
└── trends/           # 时间序列

Health（UI）= 聚合视图，不是目录
Actions（Diagnosis）= 消费 review/verify 输出
```

| 组件 | 类比 | 用户操作 |
|------|------|----------|
| Review Engine | ESLint | 项目内 **重新体检** |
| Verify | CI check | 写章/完成 Plan 后自动 |
| Health 页面 | IDE Problems 面板 | 只看 measurement 聚合 |
| Actions | Lint 解释 | 采纳 → Plan |

**禁止**：`health/` 目录塞满 review、alerts、recommendations（会变成第二个 Story OS）。

## 五、Story OS 内部分层（V3 演进，P3 先命名）

当前 Story OS 目录已混三类职责，V3 建议显式子层（**不改变现有路径**，先逻辑分层）：

```text
Story OS
├── Planner（Planning Layer）
│   ├── story_goal.json      Target State
│   ├── chapter_roadmap.json
│   ├── tasks.json
│   └── preferences.json     ← 用户可编辑
│
├── Diagnosis（Diagnosis Layer）
│   └── actions.json         ← 引擎写，用户只「采纳」
│
└ Measurement（Measurement Layer）
    ├── verify/              ← 已有
    └── health/            ← V2.8 目标（合并 review 缓存）
```

Verify 接入 Measurement 时无需再开第四套「验收中心」。

## 六、主矩阵：数据 × 写入 × 读取 × 可编辑

图例：**W**=Workspace · **K**=Knowledge · **U**=Understanding · **P**=Planner · **T**=Tasks · **A**=Actions · **M**=Measurement · **C**=Cockpit · **X**=Execute

| 数据 / 存储 | 路径或载体 | 层级 | 主要写入方 | 主要读取方 | 用户可编辑？ | 备注 |
|-------------|------------|------|------------|------------|--------------|------|
| **文稿正文** | `workspace/正文/*.md` | W | 用户、Execute | 全部 | **是** | 叙事最高真相源 |
| **大纲** | `workspace/大纲/` | W | 用户、Execute | KB scan、Planner | **是** | |
| **设定集 MD** | `workspace/设定集/` | W | 用户 | KB bootstrap | **是** | 与 knowledge 可双向启发，MD 为准 |
| **Knowledge 实体** | `knowledge/*.json` → `entities/` | Story Assets | 用户 PUT、pipeline（inferred，仅 entities） | U、P、T、Index、X | **是** | 公约 4；伏笔迁 understanding |
| **伏笔（现 foreshadows.json）** | knowledge → understanding | Intelligence | Understanding Engine | P、Review 只读 | **否** | 炸点 1 · 待迁 |
| **Story Index** | 内存 cache | 派生 | Knowledge 变更触发 rebuild | 查询 API | **否** | |
| **Summaries** | `.webnovel/summaries/` | 派生 | summaries cascade | X、prompt | **否** | 由正文重建 |
| **Understanding** | `understanding/*.json` | 派生 | Understanding Engine | P、A、M、C | **否** | 仅 `POST /sync` 触发 |
| **Story DNA** | `understanding/story_dna.json` | 派生 | 同上 | P、UI | **否** | Current State 摘要 |
| **Goal** | `planner/story_goal.json` | P · Planner | `planner/generate` | Roadmap、Tasks | **受限** | 仅 `target_state` + `success_criteria`；现状用 `current_state_ref`（炸点 2） |
| **Roadmap** | `planner/chapter_roadmap.json` | P · Planner | 同上 | Tasks decomposer | **受限** | 同上 |
| **Tasks** | `tasks/tasks.json` | P · Planner | Planner、Verify complete | Scheduler、Cockpit | **受限** | complete/start；队列 regen |
| **Preferences** | `planner/preferences.json` | P · Planner | 用户 PUT | Planner、Scheduler | **是** | 日容量、horizon 等 |
| **Actions** | `actions/actions.json` | Diagnosis | U engine、Verify followup | UI、to-plan | **否** | 采纳 → Plan |
| **Plans** | `plans/*.json` | W / Execute | 用户确认、rewrite、task/action | Execute | **是** | 怎么做 |
| **Verify log** | `verify/verify_log.json` | M | Verify | Health UI | **否** | |
| **Health snapshot** | `verify/health_snapshot.json` | M | Verify refresh | Health UI、Cockpit | **否** | → `measurement/verify/snapshots/` |
| **Review 缓存（旧）** | `studio.json` review_by_project | 废弃 | studio-service | Studio 审稿 Tab | **否** | → `measurement/review/latest.json` |
| **Versions** | `studio.json` snapshots → `versions/` | Cockpit | 用户、写章事件 | Cockpit、restore | **受限** | 创建/恢复/删，不改历史内容 |
| **studio.assets（旧）** | `studio.json` | 废弃 | 曾 Studio UI | 曾审稿 | **否** | 迁到 knowledge/ |
| **Session memory** | session 字段 | 派生 | 对话启发式 | Orchestrator | **否** | 清空会话即可 |
| **Vectors** | `data/memory/vectors.json` | 派生 | retriever indexIfStale | RAG 检索 | **否** | |
| **Planner Context** | 无独立文件 | 派生 | context-builder 运行时 | Claude prompt | **否** | KB+U+summaries 拼装 |
| **项目 meta** | `meta.json` | 系统 | 系统 | 列表、Cockpit | **受限** | 标题、归档等 |

## 七、开放问题 — 已定政策（避免未来炸点）

| 问题 | V2.8 政策 | V3 可选 |
|------|-----------|---------|
| **Goal 能编辑吗？** | 不直接编辑 JSON；改 `preferences` 或 `POST planner/generate` / `tasks/rebuild` | 表单项改 `target_state`，写 `edited_by: user` |
| **Roadmap 能编辑吗？** | 同上；禁止手改 `chapters[]` 文件 | 单章拖拽 + audit log |
| **Understanding 允许修正吗？** | 不允许改 `arcs.json`；认为 DNA 错了 → **快速/深度同步** 或改 Knowledge 后重跑 | 「标记不同意」仅影响权重，不重写文件 |
| **Planner 能改角色吗？** | **禁止**写 Knowledge；只能生成 Task/Plan 建议 | Director 提案需用户确认才 PATCH knowledge |
| **引擎推断角色与用户录入冲突？** | 同 `id` 时 **user PUT > inferred**；UI 展示 `source` | 合并 UI |
| **Health 与 studio review 并存？** | 迁移后仅 `measurement/*`；Health = 聚合 API | — |
| **Goal 含 current_state 全文？** | 禁止；仅 `current_state_ref` + `current_state_summary` | V2.8 代码已部分实现 |

## 八、写入冲突优先级（合并算法）

当同一实体（如角色 `欧阳逸清`）多处出现时：

```text
1. workspace/设定集.md + 正文     （叙事事实，扫描输入）
2. knowledge/*.json 用户 PUT      （Story Assets 权威）
3. knowledge source=inferred      （可被 2 覆盖）
4. understanding 中的分析结论      （不得反向覆盖 2）
5. actions / health 中的诊断       （不得覆盖 2）
```

**Pipeline 写入 Knowledge 时必须：**

- 设置 `source`: `user` | `inferred` | `migrated_from_studio`
- 不删除 `source=user` 的项
- 不修改用户已填的 `name`/`role`/`notes` 除非用户触发「从正文覆盖」

## 九、目标目录（Phase C 迁移后的真相源）

单项目最终只承认以下**持久化**写入根：

```text
data/projects/{id}/
├── workspace/          # W · 用户可编辑
├── knowledge/          # Story Assets · 用户可编辑（公约 4）
├── understanding/      # 派生 · 引擎写
├── planner/            # Planner 写 · preferences 用户可编辑
├── tasks/              # Planner 写 · 用户触发 regen
├── actions/            # Diagnosis 写
├── plans/              # 用户 + Execute
├── measurement/        # Measurement 写（review/verify/metrics/trends）
└── versions/           # Versions 写（自 snapshots 迁入）
```

**禁止** 新建 `health/` 数据目录；**Health** 仅为 UI/API 聚合名。见 [v2.8-measurement-layer.md](./v2.8-measurement-layer.md)。

**不应再新增：** `studio.json` 的 assets/review 写入；`data/memory` 不存放角色设定。
