# 改写指南

## 一、改写类型

### 1.1 全文重写

基于审查反馈或用户要求，重写整章正文。

**适用场景**：
- 审查报告有 blocking issue，且问题涉及整体结构
- 用户明确要求"重写这章"
- 章节与大纲偏离过大，局部修改无法修复

**执行要点**：
- 基于改写任务书全新起草
- 必须保留改写任务书中标记的伏笔连接和承接钩子
- 情节走向可以调整，但必须符合章纲的 CBN→CPNs→CEN 结构

### 1.2 局部重写

只修改特定段落、场景或对话，其余部分保持原文。

**适用场景**：
- 审查报告有 blocking issue，但问题局限在特定段落
- 用户指出"这段对话不行"或"这个场景需要改"
- AI 味集中在某些段落

**执行要点**：
- 只修改改写任务书标记的段落/场景
- 未标记部分保持原文不变
- 过渡段需要平滑衔接，不能有断裂感
- 修改后的段落风格必须与未修改部分一致

### 1.3 风格重写

保持事实、情节、对话内容不变，只改变表达方式。

**适用场景**：
- 整体 AI 味过重，需要全面去 AI 化
- 用户要求"换个风格"或"更口语化"
- 审查报告的 ai_flavor 维度大面积命中

**执行要点**：
- 事实、情节、对话内容不变
- 只改变：句式结构、用词选择、节奏控制、情绪描写方式
- 加载 `style-adapter.md` 和 `style-variants.md` 作为风格参考
- 逐段改写，确保每段的核心信息不丢失

## 二、保留约束（硬规则）

改写时必须遵守以下约束，违反任一条即为 blocking issue：

### 2.1 伏笔连接

本章涉及的 `open_loops` 中的伏笔条目必须在改写中保留：
- 埋设的伏笔：改写后仍然存在，且指向一致
- 回收的伏笔：改写后仍然被回应
- 推进的伏笔：改写后仍然有推进动作

**检查方法**：对比改写前后正文中伏笔关键词的存在性。

### 2.2 设定一致性

角色境界、地点、物品引用必须与 index.db 一致：
- 角色能力不能超越当前境界
- 地点描述不能与世界观矛盾
- 物品/货币使用必须符合已建立规则

**检查方法**：reviewer 的 setting 维度检查。

### 2.3 承接钩子

上章末尾的承诺必须在改写中回应：
- 如果上章结尾留了悬念，本章改写后仍然要回应
- 回应方式可以改变，但不能消失

**检查方法**：对比上章摘要中的钩子与改写后正文的回应。

### 2.4 章节钩子

本章末尾的悬念必须保留：
- 如果改写目的不是调整钩子，则章末悬念必须保留
- 悬念的具体内容可以改变，但"有悬念"这个状态不能消失

**检查方法**：检查改写后正文的最后 500 字是否包含未解决的问题。

## 三、版本管理

### 3.1 备份机制

改写前自动备份原文：

```
.webnovel/archive/ch{NNNN}_v{version}.md
```

- `version` 从 1 开始递增
- 每次改写自动递增版本号
- 原始版本为 `ch{NNNN}_v0.md`（首次提交时自动创建）

### 3.2 版本记录

在 `.webnovel/state.json` 的 `chapter_meta` 中记录改写历史：

```json
{
  "chapter_meta": {
    "47": {
      "rewrite_history": [
        {
          "version": 1,
          "timestamp": "2024-01-15T10:30:00Z",
          "type": "full_rewrite",
          "reason": "blocking issue: 设定矛盾",
          "issues_fixed": ["setting_001", "continuity_002"]
        }
      ]
    }
  }
}
```

### 3.3 回滚

如果改写后质量更差，可以回滚到任意历史版本：

```bash
cp .webnovel/archive/ch{NNNN}_v{version}.md 正文/第{NNNN}章-{title}.md
```

## 四、质量门控

### 4.1 改写后必须重新审查

改写完成后，必须进入标准 Step 3（审查），不能跳过。

### 4.2 改写审查重点

改写后的审查除了标准 6 个维度外，额外检查：

| 检查项 | 严重度 | 阻断 |
|--------|--------|------|
| 伏笔连接断裂 | critical | true |
| 设定引用不一致 | critical | true |
| 承接钩子消失 | high | true |
| 章节钩子消失 | high | true |
| 改写区域与未改写区域风格断裂 | medium | false |
| 字数变化超出 ±30% | low | false |

### 4.3 改写成功的标准

- 所有保留约束满足
- 标准审查无 blocking issue
- Anti-AI 终检 pass
- 改写目的达成（修复了触发改写的问题）

## 五、改写触发条件

### 5.1 用户主动触发

用户明确要求改写：
- "帮我重写第 X 章"
- "这段对话不行，改一下"
- "这章 AI 味太重，重新写"

### 5.2 审查触发

审查报告有 blocking issue，且用户选择修复：
- Step 3 审查发现 blocking issue
- Step 6 用户选择"立即修复"
- 进入改写流程，改写指令来自审查报告

### 5.3 润色触发

Anti-AI 终检 fail，且问题不是局部的：
- `anti_ai_force_check=fail`
- 问题涉及多个段落
- 进入风格重写模式
