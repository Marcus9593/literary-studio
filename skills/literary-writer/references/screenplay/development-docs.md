# 开发阶段文档写作指南

## 一、文档体系

### 1.1 电影开发文档链

```
Logline（1-2句）
  ↓
One-Pager（1页）
  ↓
Beat Sheet（2-5页）
  ↓
Treatment（10-20页）
  ↓
分场大纲（30-50页）
  ↓
剧本初稿（90-120页）
  ↓
剧本定稿
```

### 1.2 电视剧开发文档链

```
Logline + 核心概念（1页）
  ↓
Series Bible / 项目书（20-50页）
  ↓
第一季大纲（5-10页）
  ↓
分集大纲（每集1-2页）
  ↓
分场大纲（每集3-5页）
  ↓
剧本初稿
  ↓
剧本定稿
```

### 1.3 微短剧开发文档链

```
核心梗概（1页）
  ↓
人物小传（主要角色2-3页）
  ↓
分集梗概（每集3-5句话）
  ↓
前5集详细大纲
  ↓
剧本
```

## 二、Logline

### 2.1 公式

```
当[激励事件]发生时，一个[有缺陷的主角]必须[目标/行动]，否则[赌注/后果]。
```

### 2.2 要素拆解

| 要素 | 说明 | 示例（《肖申克的救赎》） |
|------|------|-------------------------|
| 激励事件 | 打破常态的事件 | 当一个银行家被冤枉入狱时 |
| 有缺陷的主角 | 身份+核心特征 | 一个冷静理性的知识分子 |
| 目标/行动 | 主角要做什么 | 必须在绝望的监狱中找到希望 |
| 赌注/后果 | 如果失败会怎样 | 否则他将在铁窗中度过余生 |

### 2.3 Logline 检查清单

- [ ] 是否在 1-2 句话内？
- [ ] 是否包含主角、目标、冲突、赌注？
- [ ] 是否让人想看这个故事？
- [ ] 是否避免了剧透高潮？
- [ ] 是否区分于同类型其他作品？
- [ ] 能否口头在 10 秒内说完？

### 2.4 Logline 示例

```
电影：
当一个失忆的特工发现自己被多方追杀时，他必须在找回记忆的同时查明谁在背后操纵一切，否则他将永远消失。

电视剧：
在一个超能力者被政府登记管控的世界里，一个没有超能力的普通人必须靠智慧在权力游戏中生存，同时隐藏自己"无能力"的秘密——因为在这个世界，无能力者比超能力者更危险。

微短剧：
当一个被公司裁员的中年男人发现妻子提出离婚的同一天，他意外获得了一部能看到每个人"人生余额"的手机——而他的余额只剩7天。
```

## 三、Treatment（故事大纲）

### 3.1 Treatment 的本质

Treatment 是用**散文体**讲述完整故事的文档——不是分点列表，是让读者像读小说一样读完后，理解整个故事的起承转合。

### 3.2 Treatment 结构

```
第一段：开场世界与主角（约1-2页）
  └─ 介绍主角的常态世界、核心缺陷、日常处境

第二段：激励事件（约半页）
  └─ 打破常态的事件，主角被迫做出选择

第三段：第一幕转折（约半页）
  └─ 主角进入新世界/新处境

第四段-第六段：第二幕发展（约5-8页）
  └─ 主角的尝试、盟友与敌人、冲突升级
  └─ 中点反转
  └─ 至暗时刻

第七段：高潮（约2-3页）
  └─ 主角如何用新认知/能力解决核心冲突

第八段：结局（约1页）
  └─ 新常态，与开场的对照
```

### 3.3 Treatment 写作要点

| 要做 | 不要做 |
|------|--------|
| 用现在时态 | 用过去时态 |
| 写角色的行动和选择 | 写角色的内心独白 |
| 保持叙事张力 | 变成流水账 |
| 包含关键对白 | 塞入完整对白段落 |
| 写出情绪变化 | 只写事件序列 |
| 散文体，可读性强 | 分点列表 |

### 3.4 Treatment 示例片段

```
李明，三十五岁，是一家小广告公司的创意总监。他每天早上七点起床，
穿同一件灰色西装，走同一条路去公司。他的创意已经三年没有被客户
通过了，但他依然每天准时出现在会议室里，用同样的话术推销同样平庸
的方案。同事们私下叫他"复印机"。

这一天，公司来了一个新客户——一家生物科技公司，要推一款"永生药"。
李明照例拿出他的模板方案，但新来的实习生张薇在会议上直接说：
"这个方案十年前就过时了。"

会议室沉默了十二秒。李明看了张薇一眼，合上了文件夹。
"那你来。"

这是李明第一次放弃。也是他最后一次。
因为张薇的方案被通过了——而李明被调去了后勤部。
```

## 四、Beat Sheet（节拍表）

### 4.1 Blake Snyder Beat Sheet (BS2)

| # | 节拍 | 位置 | 说明 |
|---|------|------|------|
| 1 | 开场画面 | 第1页 | 建立常态世界，与终场画面形成对照 |
| 2 | 主题陈述 | 第5页 | 某人说出本片主题（角色通常不理解） |
| 3 | 铺垫 | 第1-10页 | 展示主角的"缺陷世界" |
| 4 | 催化剂 | 第10页 | 激励事件 |
| 5 | 纠结/上升 | 第10-25页 | 主角犹豫、尝试、挣扎 |
| 6 | 进入第二幕 | 第25页 | 第一幕转折，主角进入新世界 |
| 7 | B故事 | 第30页 | 副线启动（通常是爱情线或友情线） |
| 8 | 游戏时间 | 第30-55页 | 故事的核心乐趣，主角在新世界探索 |
| 9 | 中点 | 第55页 | 假胜利或假失败 |
| 10 | 反派逼近 | 第55-75页 | 压力加大，反派势力增强 |
| 11 | 一无所有 | 第75页 | 至暗时刻 |
| 12 | 灵魂黑夜 | 第75-85页 | 主角最绝望的时刻 |
| 13 | 进入第三幕 | 第85页 | 主角找到解决方案 |
| 14 | 结局 | 第85-110页 | 高潮 + 余波 |
| 15 | 终场画面 | 第110页 | 新常态，与开场画面对照 |

### 4.2 微短剧 Beat Sheet

| # | 节拍 | 位置 | 说明 |
|---|------|------|------|
| 1 | 钩子 | 0-5秒 | 吸引注意力的画面/台词/冲突 |
| 2 | 建置 | 5-20秒 | 快速交代人物和处境 |
| 3 | 冲突 | 20-60秒 | 核心矛盾展开 |
| 4 | 升级 | 60-90秒 | 冲突加剧或反转 |
| 5 | 集尾钩子 | 最后5-10秒 | 悬念/危机/反转 |

## 五、Series Bible（剧集项目书）

### 5.1 Series Bible 结构

```
一、项目概述（1-2页）
  ├── Logline
  ├── 类型/调性
  ├── 目标平台
  ├── 目标观众
  ├── 集数×时长
  └── 核心卖点

二、世界观（3-5页）
  ├── 时代背景
  ├── 核心设定
  ├── 规则与限制
  └── 视觉风格参考

三、人物谱系（5-10页）
  ├── 核心角色（每人1-2页）
  ├── 重要配角（每人半页）
  └── 关系图谱

四、季线规划（3-5页）
  ├── 第一季核心冲突
  ├── 季线弧光
  ├── 关键转折点
  └── 季终悬念

五、分集梗概（每集半页-1页）
  ├── 第1集
  ├── 第2集
  └── ...

六、后续季展望（1-2页）
  ├── 第二季方向
  └── 系列终极走向

七、附录（可选）
  ├── 术语表
  ├── 世界观设定图
  └── 参考作品分析
```

### 5.2 核心角色描述模板

```
【角色名】

基本信息
  年龄：  |  职业：  |  身份：

外貌特征（选角参考）
  ___

性格关键词
  1. ___  2. ___  3. ___

核心欲望（Want）
  表层目标：___

核心需求（Need）
  深层需求：___（通常与欲望矛盾）

最大恐惧
  ___

说话风格
  ___

标志性动作/习惯
  ___

秘密
  ___

角色功能
  ___

弧光
  从___变成___

与其他角色的关系
  与A：___
  与B：___
```

### 5.3 分集大纲模板

```
第X集：《集名》

核心事件：（一句话概括本集最重要的事）

A故事：（主线发展）
___

B故事：（副线发展）
___

角色发展：（本集角色变化）
___

集尾钩子：（让观众想看下一集的悬念）
___

客串/新角色：___
关键场景：___
```

## 六、改稿与应对 Notes

### 6.1 理解 Notes 的本质

制片人/导演/平台给的 note 不是"命令"，是**症状描述**。你要诊断背后的病因。

```
Note："节奏太慢"
  可能的病因：
  ├── 信息密度不够 → 合并场景
  ├── 冲突不够强 → 加强对抗
  ├── 铺垫太长 → 删减过渡
  └── 对白太啰嗦 → 精简台词

Note："角色不讨喜"
  可能的病因：
  ├── 缺陷太重 → 加"救猫咪"场景
  ├── 动机不清 → 补角色背景
  ├── 太被动 → 给角色更多主动选择
  └── 缺乏魅力点 → 加一个特长或幽默感
```

### 6.2 常见 Notes 应对策略

| Note | 背后问题 | 应对 |
|------|---------|------|
| "节奏太慢" | 信息密度/冲突不足 | 合并场景、删过渡、提前冲突 |
| "角色不讨喜" | 共情点缺失 | 加救猫咪场景、展示合理动机 |
| "看不懂" | 信息传递不清 | 补锚点场景、简化线索链 |
| "太平了" | 缺少转折 | 中点加反转、增加意外元素 |
| "结尾不满意" | 高潮与主题脱节 | 检查高潮是否回应核心需求 |
| "投资太大" | 制作成本超预算 | 合并场景、减少外景、压缩群演 |
| "跟XX太像" | 差异化不足 | 找到独特角度、改变核心梗 |
| "不够商业化" | 观众期待未满足 | 检查类型承诺是否兑现 |
| "太商业化" | 艺术表达不足 | 增加角色深度、减少套路 |
| "集与集之间太平" | 缺乏连续钩子 | 每集末尾加悬念 |

### 6.3 改稿原则

1. **先理解，后动手**——不要拿到 note 就改，先分析背后的真正问题
2. **保护核心**——主题、主角弧光、核心冲突是底线，其他都可以动
3. **先大后小**——先解决结构问题，再调对白细节
4. **不要恋旧**——"我写得很辛苦"不是保留某场戏的理由
5. **留备份**——每次大改前保存当前版本
6. **确认方向**——改之前先和制片人/导演确认你的理解是否正确

### 6.4 改稿流程

```
收到 Notes
  ↓
分析：Note 背后的真正问题是什么？
  ↓
列出修改方案（至少2个方向）
  ↓
与制片人/导演确认方向
  ↓
先改结构（场景增删合并）
  ↓
再改节拍（场景内部调整）
  ↓
再改对白（台词打磨）
  ↓
通读检查一致性
  ↓
交付修改稿
```

## 七、剧本朗读会

### 7.1 什么是剧本朗读会

剧本朗读会（Table Read）是让演员围坐在一起朗读剧本的会议。目的是：
- 发现对白是否自然
- 检验节奏是否合理
- 发现逻辑漏洞
- 评估角色声音是否区分明显

### 7.2 编剧在朗读会的角色

- **听，不要辩解**——如果演员读不顺，是剧本的问题
- **记录问题点**——在剧本上标记卡壳的地方
- **关注反应**——哪里有笑声？哪里有沉默？哪里有人走神？
- **不要当场改**——会后统一修改

### 7.3 朗读会后的修改重点

- 任何读不顺的对白都需要改
- 任何引发意外反应（笑/困惑）的地方都需要检查
- 任何超过2分钟没有变化的段落都需要压缩
